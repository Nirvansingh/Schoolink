# Google Gemini API Integration (Final Phase Only)

**Important:** Do **not** integrate the Google Gemini API immediately.

The AI features should first be developed using placeholder functions, dummy data, and a modular architecture. This allows the application's UI, workflow, and logic to be fully completed and tested before connecting to the real AI service.

The Google Gemini API should be integrated only in the **final phase of development** for the following reasons:

* Prevent exposing the API key during early development.
* Avoid unnecessary API usage while the application is still changing.
* Reduce development costs and API requests.
* Make debugging easier by separating application logic from AI functionality.
* Ensure the project architecture is complete before connecting external services.

## AI Architecture

Build the AI system so that all AI requests are handled through a dedicated module (for example, `ai.js`).

For now:

* Create placeholder functions that return mock AI responses.
* Ensure every AI feature calls these placeholder functions.
* The rest of the application should not depend directly on the Gemini API.

When development is complete:

* Replace the placeholder functions with real Google Gemini API requests.
* The rest of the application should continue working without modification.

## API Security

The Google Gemini API key must **never** be hardcoded directly into production code or committed to GitHub.

For development, use a temporary configuration file if necessary.

For deployment (such as on Vercel), store the API key using **Environment Variables** so it remains private and secure.

Design the code so switching from a development API key to a production environment requires minimal changes.

The objective is to keep the project secure, modular, and easy to maintain while following modern software development practices.
